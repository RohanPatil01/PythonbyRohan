Category 1 of 6

🔢 Numeric Types: int, float, complex

int stores whole numbers with no decimal. No size limit. Use underscores for readability like writing 1,000,000 in real life.

python
age    = 24
score  = -5
big    = 1_000_000   # same as 1000000
binary = 0b1010      # base 2 = 10
hexa   = 0xFF        # base 16 = 255
print(type(age))     # <class 'int'>

float stores decimal numbers. Supports scientific notation. Watch out for precision issues.

python
temperature = 36.6
pi   = 3.14159
sci  = 1.5e3           # 1500.0
print(0.1 + 0.2)        # 0.30000000000000004
print(round(0.1+0.2,2)) # 0.3 (use round() to fix)
📝 Why 0.1 + 0.2 gives 0.30000000000000004: Computers store numbers in binary. Some decimal fractions cannot be stored exactly in binary, just like 1/3 cannot be written exactly in decimal (0.333...). This is not a Python bug. Use round() or the decimal module when precision matters.

complex:real and imaginary parts. Used in engineering. Rarely needed at beginner level.

python
z = 3 + 4j
print(z.real)   # 3.0
print(z.imag)   # 4.0
print(type(z))  # <class 'complex'>
Category 2 of 6

✅ Boolean: True and False

Boolean holds one of two values: True or False. Think of a light switch. Either on or off. Booleans are secretly integers in Python: True equals 1 and False equals 0.

python
is_student = True
eligible   = 10 > 5      # True (comparison gives bool)
print(True + True)       # 2
print(False + 5)         # 5
Important Concept

💡 Truthy and Falsy Values

Every Python value has a hidden boolean meaning. When Python evaluates a condition, it silently converts the value to True or False. This is called truthiness. Knowing this saves you from subtle bugs.

Falsy (treated as False)

  • False
  • None
  • 0 (int)
  • 0.0 (float)
  • "" (empty string)
  • [] (empty list)
  • {} (empty dict)
  • () (empty tuple)
  • set() (empty set)

Truthy (treated as True)

  • True
  • Any non-zero number
  • " " (space, NOT empty)
  • "False" (non-empty string)
  • [0] (list with one item)
  • Any non-empty collection
python
# Edge cases that surprise beginners
print(bool(""))        # False  (empty string)
print(bool(" "))       # True   (space = non-empty string!)
print(bool("False"))   # True   (non-empty string, even if it says False)
print(bool("0"))       # True   (non-empty string, not the integer 0)
print(bool(0))         # False  (actual integer zero)
print(bool([0]))       # True   (list with one item)
print(bool([]))        # False  (empty list)
💡 This is why if my_list: works as a non-empty check. Python converts the list to bool automatically. No need for if len(my_list) > 0:.
Keywords

🔑 True, False, and None are Reserved Keywords

True, False, and None are not just values. They are reserved keywords in Python. You cannot use them as variable names, and capitalisation is strict.

python
# These raise SyntaxError
# True = 10   |  False = 0   |  None = "hi"

# Capitalisation is strict
print(True)      # works
# print(true)   # NameError
# print(TRUE)   # NameError

# None is a singleton:only one exists in memory
print(id(None) == id(None))  # True always
📝 Since None is a singleton, always use x is None instead of x == None. The is operator checks memory identity, which is the correct and faster check for None.
Category 3 of 6

🚫 NoneType: The Absence of a Value

None means nothing is here. Not zero. Not False. Not empty string. It is the complete absence of any value. Think of an empty field on a form that has not been filled yet.

python
result = None
print(type(result))    # <class 'NoneType'>

if result is None:    # correct way to check
    print("No value yet")

# Functions that return nothing give None
def greet():
    print("Hello")

x = greet()
print(x)              # None
Category 4 of 6

📋 Sequence Types: str, list, tuple, range

Sequence types store multiple items in order. Every item has a position (index starting at 0). We go deep on each in dedicated topics. Here is an introduction to all four.

python
name    = "Rohan Patil"            # str: immutable
friends = ["Siddhesh", "Sagar"]  # list: mutable
dob     = (11, 12, 2001)       # tuple: immutable
r       = range(5)              # range: 0,1,2,3,4
print(name[0])                  # R
print(list(r))                  # [0,1,2,3,4]
Category 5 of 6

🗂️ Mapping Type: dict

A dictionary stores key-value pairs. Think of a contacts app: search by name (key), get the phone number (value). Not indexed by position, indexed by key.

python
student = {
    "name"   : "Rohan Patil",
    "college": "B. N. Bandodkar College of Science",
    "cgpa"   : 10.0
}
print(student["name"])   # Rohan Patil
Category 6 of 6

🔵 Set Types: set and frozenset

A set is an unordered collection of unique items. No duplicates allowed. Great for removing duplicates and fast membership checks. frozenset is the immutable version:covered in one line.

python
subjects = {"Python", "ML", "Stats", "Python"}
print(subjects)   # {'Python','ML','Stats'}:duplicate removed
fs = frozenset([1,2,3])  # immutable set
Common Confusion

⚠️ Empty Set vs Empty Dictionary: set() vs {}

This is one of the most common Python traps. Both sets and dicts use curly braces. But {} always creates a dictionary, not a set. Python chose dict as the default because dicts came before sets historically.

python
x = {}             # dict, NOT a set
print(type(x))     # <class 'dict'>

y = set()          # correct empty set
print(type(y))     # <class 'set'>

z = {1, 2, 3}      # non-empty set (values, no colon) = set
d = {"a": 1}       # key:value (colon) = dict
⚠️ Always use set() for an empty set. Never use {}.

Core Concept

🔒 Mutable vs Immutable

Every data type is either mutable or immutable. This single concept explains a lot of confusing Python behaviour.

Immutable: the object cannot be changed after creation. If you try to "change" it, Python creates a brand new object in memory. Mutable: the object can be changed in place:same object, same memory address.

CategoryTypesChanges after creation?Hashable (dict key)?
Immutableint, float, complex, bool, str, tuple, frozenset, NoneTypeNoYes
Mutablelist, dict, set, bytearrayYesNo

Use id() to prove it. If the id stays the same after a change, the object was mutated in place. If the id changes, a new object was created.

python
# IMMUTABLE: string
name = "Rohan"
print(id(name))          # e.g. 140234001
name = name + " Patil"
print(id(name))          # DIFFERENT id:new object

# MUTABLE: list
friends = ["Siddhesh", "Sagar"]
print(id(friends))       # e.g. 140234999
friends.append("Rohan")
print(id(friends))       # SAME id:modified in place
💡 Why this matters in real code: when you pass a list to a function and the function modifies it, your original list changes too (same mutable object). When you pass a string or int, the original is always safe.

🪆 Nested Data Structures

Python lets you put data structures inside other data structures. A list can contain dicts. A dict can hold lists as values. This is how real-world data looks:like a JSON response from an API or a database record.

    "name"    : "Rohan Patil",
    "subjects": ["Python", "ML", "Stats"],

}
print(student["subjects"][0])  # Python
Type Checking

🔍 type() vs isinstance(): Know the Difference

You know type(). Now meet isinstance(). Both check types but they work differently and one is almost always better in real code.

python
x = 42
print(type(x))                    # <class 'int'>
print(type(x) == int)             # True (exact match)

print(isinstance(x, int))         # True
print(isinstance(x, (int,float))) # True (check multiple types)

# Key difference: bool is a subclass of int
print(type(True) == int)           # False (exact type is bool)
print(isinstance(True, int))       # True  (bool IS a subclass of int)

Type Casting

🔁 Type Casting: Implicit and Explicit

Implicit casting happens automatically when Python combines compatible types.

python
z = 5 + 2.0     # int promoted to float automatically
print(z)         # 7.0
print(type(z))   # <class 'float'>
print(10 + True) # 11

Explicit casting is when you manually convert using built-in functions.

python
int("24")        # 24
int(9.9)         # 9  (truncates, does NOT round)
float("3.14")    # 3.14
str(100)         # "100"
list((1,2,3))   # [1, 2, 3]
set([1,2,2])    # {1, 2}

Yes always works. Only if numeric works for "24" but raises ValueError for "hello".
Errors to Know

🚨 Type Casting Errors and ValueError

Not every conversion is possible. When you try to convert something that cannot be converted, Python raises an error. Knowing which error and why helps you debug faster.

python
# ValueError: right type, wrong value
int("hello")          # ValueError: invalid literal for int()
float("twenty")       # ValueError
int("3.14")           # ValueError: can't convert float string to int directly
int(float("3.14"))    # 3:correct: float first, then int

# TypeError: incompatible type entirely
int([1,2,3])          # TypeError: can't convert list to int
int(None)              # TypeError
⚠️ ValueError vs TypeError: ValueError means the type is right but the value cannot be converted (like "hello" to int). TypeError means the type itself is wrong for the operation (like converting a list to int). The common scenario: user types "twenty" when you do int(input(...)). You will handle this with try-except in Exception Handling.
Most Common Pattern

⌨️ input() with Type Casting: The Pattern You Will Use Every Day

input() always returns a string. You almost always need to convert it. Here is the complete guide to every input pattern you will need.

python
name    = input("Name: ")                      # str
age     = int(input("Age: "))                  # int
cgpa    = float(input("CGPA: "))               # float
is_pass = input("Passed? (yes/no): ") == "yes" # bool via comparison

# Multiple values on one line
a, b = map(int, input("Two numbers: ").split())
print(a + b)

# List of numbers from one line
marks = list(map(int, input("Marks: ").split()))
print(marks)       # [85,90,78] if user typed "85 90 78"
print(sum(marks))  # 253
💡 The map(int, input().split()) pattern is extremely common. It converts all items in a space-separated input to integers in one line. Keep this in your toolkit.

Beginner Level
Q1. What are the built-in data types in Python? How are they categorized? Beginner ›
One-line answer
Python has 6 categories of built-in types: Numeric, Boolean, Sequence, Mapping, Set, and NoneType. You never declare them explicitly. Python infers the type from the value.
  • Numeric: int, float, complex
  • Boolean: bool (True or False, subclass of int)
  • Sequence: str, list, tuple, range
  • Mapping: dict (key-value pairs)
  • Set: set (unique, unordered), frozenset (immutable set)
  • NoneType: None (the absence of a value)

Use type(x) to check the type of any value at runtime.

⚡ Say it in an interview
Python has 6 built-in type categories. The most commonly used are int, float, str, list, dict, set, bool, and None. Python is dynamically typed so the type is inferred from the value, not declared upfront.
Q2. What is the difference between int and float? Beginner ›
One-line answer
int is for whole numbers with exact precision. float is for decimals but uses IEEE 754 binary representation, which introduces tiny rounding errors.
  • int: no decimal point, unlimited precision in Python (no overflow like C). Example: x = 9999999999999999999 works fine.
  • float: has a decimal point, limited to about 15-17 significant digits, stored in 64-bit IEEE 754 format. This causes the classic:
0.1 + 0.2  # gives 0.30000000000000004, not 0.3

When exact decimal precision matters (like money), use the decimal module or round() for display.

Common trap
Never compare floats with ==. Use abs(a - b) < 1e-9 or math.isclose(a, b) instead. Direct equality checks on floats will silently fail due to precision errors.
Q3. What is None in Python? How is it different from 0, False, or ""? Beginner ›
One-line answer
None represents the complete absence of a value. It is not zero, not False, not an empty string. Those are actual values. None means "nothing here at all."

None belongs to its own type, NoneType, and is a singleton: only one None object exists in all of Python's memory.

  • 0 is an integer with value zero
  • False is a boolean
  • "" is an empty string
  • None is the absence of any value. A function that does not explicitly return something returns None.

All four are falsy in a boolean context, but they are very different objects with different types.

⚡ Say it in an interview
Always check for None with x is None, not x == None. Because None is a singleton, identity check (is) is both more correct and slightly faster than equality check (==).
Q4. What is the difference between a list and a tuple? Beginner ›
One-line answer
List is mutable (changeable), tuple is immutable (fixed). Use a list when data will change, a tuple when data should stay constant.
  • list: [1, 2, 3], mutable, supports .append(), .remove(), etc. Slightly slower and uses more memory.
  • tuple: (1, 2, 3), immutable, no modification after creation. Faster iteration, smaller memory footprint.

A key consequence of immutability: tuples are hashable, so they can be used as dictionary keys. Lists cannot.

Good real-world split: use a list for a shopping cart (changes constantly), use a tuple for GPS coordinates or an RGB colour value (should never change).

⚡ Say it in an interview
The main differences are mutability and hashability. Tuples are immutable, faster, and can be used as dict keys. Lists are mutable and flexible. Choose based on whether the data should change.
Intermediate Level
Q5. What is the difference between type() and isinstance()? Intermediate ›
One-line answer
type() returns the exact type. isinstance() checks the type and all its parent classes. Use isinstance() in real code.

The critical difference shows up with inheritance. Since bool is a subclass of int:

type(True) == int       # False (exact type is bool)
isinstance(True, int)   # True  (bool is a subclass of int)

isinstance() also accepts a tuple of types for checking multiple at once:

isinstance(x, (int, float))  # True if x is either
⚡ Rule
Use type() for debugging when you want to see exactly what something is. Use isinstance() in production code for validation, because it respects inheritance and is more flexible.
Q6. What are truthy and falsy values? Name some surprising ones. Intermediate ›
One-line answer
Every Python value is either truthy or falsy. Falsy values are: None, False, 0, 0.0, "", [], {}, (), set(). Everything else is truthy.

Python evaluates truthiness by calling bool(x) internally. The surprising cases interviewers love to ask about:

  • " " (a single space): truthy. The string is not empty, it contains a space character.
  • "False": truthy. It is a non-empty string. The content does not matter, only the length.
  • [0]: truthy. The list has one item. Python checks whether the container is empty, not what is inside it.
  • 0.0: falsy. Zero is zero regardless of type.
Common trap
Assuming "False" is falsy. It is not. It is a non-empty string and evaluates to True in a condition. Only the boolean value False and the empty string "" are falsy.
Q7. What is type casting? What is the difference between ValueError and TypeError? Intermediate ›
One-line answer
Type casting converts a value from one type to another. ValueError means the value is wrong for that conversion. TypeError means the type itself cannot be used that way at all.

Two kinds of type casting:

  • Implicit: Python does it automatically. int + float gives float without you doing anything.
  • Explicit: you call int(), float(), str(), bool() yourself.

The error distinction:

  • int("hello"): raises ValueError. The type (str) is fine to cast from, but the value "hello" cannot be converted to an integer.
  • int([1, 2, 3]): raises TypeError. You cannot cast a list to int at all. The operation itself is invalid.
⚡ Quick distinction
ValueError: right type, wrong value. TypeError: wrong type entirely. Knowing this difference helps you write better try/except blocks.
Advanced Level
Q8. What is the difference between mutable and immutable types? How does id() reveal this? Advanced ›
One-line answer
Mutable objects can be changed in place. Immutable objects cannot. id() exposes this: modifying a mutable object keeps the same id, while "modifying" an immutable one gives you a new object with a different id.

Mutable: list, dict, set. You can add, remove, or update elements and the object's identity (memory address) stays the same.

Immutable: int, float, str, tuple, frozenset, bool, NoneType. Any operation that appears to change them actually creates a new object.

lst = [1, 2]
print(id(lst))     # e.g. 140512345
lst.append(3)
print(id(lst))     # same id — same object, modified in place

s = "hello"
print(id(s))       # e.g. 140512999
s += " world"
print(id(s))       # different id — new object was created

Immutable objects are hashable and can be used as dict keys. Mutable objects cannot.

⚡ Say it in an interview
Mutability is about whether you can change the object in place. id() proves it: same id after change means mutable, different id means a new object was created (immutable).
Q9. Why does {} create a dict and not a set in Python? Advanced ›
One-line answer
Dictionaries existed in Python long before sets. By the time sets were added, {} was already dict syntax. To avoid breaking code, it stayed that way.

The rules today:

  • {}: creates an empty dict, not a set
  • {1, 2, 3}: creates a set (non-empty curly braces with no colons)
  • {"a": 1}: creates a dict (key-value pairs with colons)
  • set(): the only way to create an empty set

This is one of those historical quirks in the language. Knowing it well makes you look sharp in an interview.

Common trap
Writing empty_set = {} when you meant to create a set. You actually created a dict. Always use set() for an empty set.
Q10. Why are True, False, and None capitalised in Python? Advanced ›
One-line answer
They are reserved keywords and singletons. Python's convention capitalises constants, and the uppercase form was chosen to make them visually distinct and prevent accidental shadowing.

True, False, and None are keywords, not just global variables. You cannot assign to them: True = 1 raises a SyntaxError.

They are also singletons: only one instance of each exists in Python's memory at any time. This is why identity checks work reliably:

  • x is None is always correct because there is exactly one None object
  • x is True works the same way

Historically, in Python 2, True and False were just variables that could be reassigned (True = False was legal). Python 3 made them proper keywords to prevent this chaos.

⚡ Say it in an interview
They are keywords and singletons. Capitalisation follows Python's constant convention and makes them visually distinct from user-defined names. In Python 3 they cannot be reassigned.
💡 Data type mistakes are sneaky. Python often does not raise an error immediately. The bug shows up later when you try to use the wrong type in a calculation or comparison.
1
Using == None instead of is None
Logic Error
›
Why it happens: == checks equality, and is checks identity (same object in memory). For None, you always want identity. Custom classes can override == to return anything, which can silently fool you. is None is guaranteed to be correct.
❌ Not recommended
python
if x == None:
    print("no value")
Why this is risky
Custom classes can override ==
to return True even when x is not None.
✅ Correct
python
if x is None:
    print("no value")

if x is not None:
    print("has value")
🧠 Memory Trick
There is only one None in all of Python. It is a singleton. So you are not asking "is this equal to None?" you are asking "is this the exact same None?" That is an identity check, which is what is does.
⚡ Key Rule
Always use is None or is not None. PEP 8 (Python's official style guide) explicitly requires this. Many linters will warn you if you use == None.
2
Expecting int() to round instead of chop
Logic Error
›
Why it happens: The name int() sounds like it converts to the nearest integer. It does not. It truncates (chops off) everything after the decimal, always moving towards zero. So int(9.9) gives 9, not 10.
❌ Wrong assumption
python
print(int(9.9))   # expected 10, got 9
print(int(-9.9))  # expected -10, got -9
Actual output
9
-9
(truncated towards zero, not rounded!)
✅ Use round() for rounding
python
print(round(9.9))   # 10 (proper rounding)
print(round(9.4))   # 9
print(int(9.9))     # 9 (use only to drop decimal)
🧠 Memory Trick
Think of int() as a pair of scissors that cuts off the decimal tail. It does not care how long the tail is. 9.9999 becomes 9, not 10. If you want rounding, use round(), which actually looks at the value.
⚡ Key Rule
int() truncates (chops). round() rounds to nearest. Use int() only when you deliberately want to drop the decimal part.
3
Thinking "False" or " " (a space) are falsy
Logic Error
›
Why it happens: You know False is falsy and assume the string "False" is too. But Python does not read the content of a string. It only asks one question: is this string empty? Any string with even one character (including a space) is truthy.
❌ Wrong assumption
python
print(bool("False"))  # True, not False!
print(bool(" "))      # True, space counts!
Actual output
True
True
(both are non-empty strings, so truthy)
✅ What is actually falsy
python
print(bool(""))        # False (empty string)
print(bool(False))     # False (actual boolean)
print(bool(0))         # False (zero)
🧠 Memory Trick
Python is like a bouncer who only checks if the bag is empty, not what is written on it. A bag with "empty" written on it is still not an empty bag. "False" has 5 characters in it, so Python says: not empty, truthy.
⚡ Key Rule
Only the empty string "" is falsy. Any string with at least one character, even a space " ", is truthy. Python checks emptiness, not content.
4
Using {} to create an empty set
Logic Error
›
Why it happens: {} looks like it should be an empty set, but Python decided early on that {} means an empty dict. There is no error, no warning. Your variable silently becomes the wrong type and causes confusion later.
❌ Wrong
python
s = {}
print(type(s))
Actual output
<class 'dict'>
(not a set! silent wrong type)
✅ Correct
python
s = set()
print(type(s))  # <class 'set'>

# set with items: curly braces work here
s2 = {"a", "b", "c"}
🧠 Memory Trick
Curly braces {} were claimed by dicts before sets existed in Python. So Python gave sets the function set(). Remember: empty set always needs the word "set" in it. If you just write {}, Python assumes dict.
⚡ Key Rule
Use set() for an empty set. Use {} for an empty dict. Use {"a", "b"} for a set with items. The only safe empty set syntax is set().
5
int("3.14") raises a ValueError
ValueError
›
Why it happens: int() can convert the string "3" because it looks exactly like a whole number. But "3.14" has a decimal point, and int() does not know how to handle that. It gives up and raises a ValueError.
❌ Wrong
python
int("3.14")
Python says
ValueError: invalid literal for int()
with base 10: '3.14'
✅ Two-step conversion
python
int(float("3.14"))  # 3
# string to float first, then chop to int
🧠 Memory Trick
int() is strict: it only accepts strings that look exactly like whole numbers. If there is a dot in the string, it refuses. float() is more relaxed and accepts both. So the two-step route is: string to float, then float to int.
⚡ Key Rule
To convert a decimal string to int, use int(float("3.14")). Never pass a decimal string directly to int().
6
Using float for money calculations
Logic Error
›
Why it happens: Floats store numbers in binary (base 2), and most decimal fractions like 0.1 cannot be represented exactly in binary. The tiny rounding errors are invisible in small calculations but accumulate into real problems in financial code.
❌ Wrong for money
python
print(0.1 + 0.2)
Actual output
0.30000000000000004
(not 0.3 — floats have binary precision limits)
✅ Use Decimal for money
python
from decimal import Decimal
total = Decimal("0.1") + Decimal("0.2")
print(total)  # 0.3 (exact)
🧠 Memory Trick
Floats are great scientists (lots of precision, tiny errors acceptable) but terrible accountants (errors compound, every paisa matters). When dealing with money, use Decimal, which works in base 10 exactly like humans count.
⚡ Key Rule
For money or any calculation where exactness matters, use from decimal import Decimal and always pass values as strings: Decimal("0.1"), not Decimal(0.1).
Key Takeaways
🔢Numeric typestypes
  • int for whole numbers, float for decimals, complex for j notation
  • float has binary precision limits: 0.1 + 0.2 is not exactly 0.3
  • Use round() to control decimal display
x = 42 | y = 3.14 | z = 3+4j
✅Booleantype
  • Only two values: True and False
  • Secretly an int: True == 1, False == 0
  • Every value in Python has a truthy or falsy nature
bool(0) # False | bool("hi") # True
🚫NoneTypetype
  • None means "no value" — not 0, not False, not an empty string
  • Only one None exists in all of Python (singleton)
  • Always check with is None, never with == None
if x is None: ... # correct
📋Sequencestypes
  • str, list, tuple, range: ordered and indexable
  • list is mutable (you can change it); tuple is not
  • Strings are immutable: every change creates a new string
li = [1,2,3] | tu = (1,2,3)
🗂️dictmapping
  • Stores key-value pairs, not indexed by position
  • Keys must be unique and immutable (str, int, tuple)
  • Values can be anything, including other dicts
d = {"name": "Rohan", "age": 21}
🔵setcollection
  • Unordered collection of unique items
  • set is mutable; frozenset is not
  • Empty set must be set(), never {} (that's a dict)
s = {1, 2, 3} | empty = set()
🔒Mutable vs Immutableconcept
  • Mutable: list, dict, set — same object changes in place
  • Immutable: str, tuple, int, float — any change creates a new object
  • Check with id() to see if it is the same object
id(x) before vs after tells the story
🔍Type checkingconcept
  • type(x) returns the exact class, good for debugging
  • isinstance(x, int) is better in real code: handles subclasses
  • Can check multiple types: isinstance(x, (int, float))
isinstance(x, (int, float)) # True for both
🔁Implicit castingconcept
  • Python automatically widens types when mixing them
  • int + float gives float, not int
  • Python never silently narrows (no accidental data loss)
3 + 1.0 # 4.0, not 4
⚡Explicit castingconcept
  • int() truncates (chops), not rounds
  • int("3.14") raises ValueError — go through float first
  • round() is separate from int()
